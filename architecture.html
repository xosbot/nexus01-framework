System Architecture

A minimal, modular framework built for clarity — not complexity

🎯

Minimal by Design

Every component exists because it solves a real problem. No abstraction layers, no "just in case" features. If you can't explain why it's there, it shouldn't be.

🔌

Zero-Config Local

Run the entire stack on your laptop with no API keys, no Docker, no cloud services. SQLite, Ollama, and asyncio — that's it.

Interfaces

Telegram Bot · Rich CLI · Web Dashboard
▸
What It Does

User-facing entry points. Handles input parsing, output formatting, and session context. All three interfaces share the same underlying agent bus — switch from Telegram to CLI without changing a single agent.

Components
  • Telegram Bot — python-telegram-bot in webhook mode. Handles /start, /help, and free-form messages. Supports inline keyboards for Cold Mode approval.
  • Rich CLI — Interactive prompt with rich library. Syntax-highlighted output, progress spinners, and interactive agent selection.
  • Web Dashboard — FastAPI + HTMX (planned). Real-time agent status, conversation history, and configuration UI.
Connections

Below: Sends user messages to the Orchestration layer via asyncio.Queue. Receives formatted responses back.

# Interface → Orchestration handoff async def handle_message(msg: str): await message_queue.put({ "source": "telegram", "text": msg, "user_id": ctx.user_id, "timestamp": datetime.now() })

Orchestration

LangGraph Patterns · Cold Mode Protocol · Timing Engine
▸
What It Does

The traffic controller. Routes tasks to the right agent, manages state transitions, enforces Cold Mode gates, and tracks timing/latency across the entire pipeline.

Components
  • Agent Router — Pattern matching on intent. Classifies messages into categories (research, execute, analyze, chat) and routes accordingly.
  • Cold Mode Protocol — Permission gates before destructive actions. No file writes, no API calls, no code execution without explicit user approval.
  • Timing Engine — Tracks request→response latency per agent, per task type. Feeds into performance dashboards and optimization decisions.
  • State Machine — Agent lifecycle: idle → working → complete → error. Each transition is logged and observable.
Connections

Above: Receives from Interfaces. Below: Dispatches to Agent Layer. Side: Consults Knowledge layer for context.

# Cold Mode gate — blocks until user approves async def cold_mode_check(action: str) -> bool: if action in DESTRUCTIVE_ACTIONS: approved = await prompt_user_approval(action, timeout=30) if not approved: log.warning("Cold mode rejected", action=action) return False return True

Agent Layer

OSINT Agent · Executor Agent · Analyst Agent
▸
What It Does

Three specialized agents, each with a single responsibility. They don't talk to each other directly — they communicate through the message bus. This isolation makes debugging and testing straightforward.

Components
  • OSINT Agent — Gathers intelligence, scrapes data, fetches signals from the web. Uses search APIs and web scrapers to collect raw information.
  • Executor Agent — Runs code, writes files, makes API calls. The "hands" of the system. All I/O passes through Cold Mode first.
  • Analyst Agent — Analyzes patterns, generates summaries, makes decisions. The "brain" that synthesizes information from OSINT and Executor outputs.
Connections

Above: Receives tasks from Orchestration. Below: Uses Communication layer (Queue + A2A) to exchange messages. Side: Reads/writes Knowledge layer for context.

# Agent base class — every agent extends this class BaseAgent: def __init__(self, name: str, tools: list): self.name = name self.tools = tools self.memory = [] async def run(self, task: str) -> dict: # Think → Act → Observe loop thought = await self.think(task) result = await self.act(thought) self.memory.append({"task": task, "result": result}) return result

Communication

asyncio.Queue · A2A Protocol · MCP
▸
What It Does

The nervous system. Routes messages between agents, tools, and external services. Handles both internal (asyncio.Queue for in-process) and external (A2A/MCP for cross-system) communication.

Components
  • asyncio.Queue — In-process message bus for agent-to-agent communication. Zero dependencies, instant message passing.
  • A2A Protocol — Agent-to-Agent for external agent discovery. Enables NEXUS-01 to communicate with other agent systems (planned).
  • MCP — Model Context Protocol for tool integration. Standard interface for connecting to external tools and services (planned).
Connections

Above: Agents publish/subscribe to message topics. Below: Knowledge layer persists messages for recall. Side: Tools register their capabilities here.

# Message bus — pub/sub pattern message_queue = asyncio.Queue() async def publish(topic: str, data: dict): await message_queue.put({ "topic": topic, "timestamp": time.time(), **data }) async def subscribe(topic: str): while True: msg = await message_queue.get() if msg["topic"] == topic: yield msg

Knowledge

SQLite Memory · ChromaDB Vectors · Embeddings
▸
What It Does

Persistent memory. Stores structured data (SQLite), vector embeddings (ChromaDB), and handles semantic search over past interactions. This is what gives NEXUS-01 "memory" — it remembers what it learned.

Components
  • SQLite — Conversations, tasks, metadata, agent state. Single file, zero config, portable across machines.
  • ChromaDB — Vector store for semantic search. Enables "find similar past tasks" and "recall relevant context" queries.
  • Embeddings — sentence-transformers for local embedding generation. No API calls needed, runs entirely on-device.
Connections

Above: Agents read/write via Knowledge API. Below: Tools layer feeds raw data in. This layer is the shared memory across all agents.

# Semantic memory lookup async def recall(query: str, k: int = 5): embedding = model.encode(query) results = collection.query( query_embeddings=[embedding.tolist()], n_results=k ) return results["documents"][0]

Tools

Web Scraper · Shell Executor · Search API
▸
What It Does

External world access. Each tool is a thin wrapper around a library. Tools are registered in a registry and agents select which to use per task. No implicit tool access — agents must explicitly request tools.

Components
  • Web Scraper — httpx + BeautifulSoup4. Async HTTP with HTML parsing. Rate-limited and respects robots.txt.
  • Shell Executor — subprocess with sandboxing. Runs commands in a restricted environment with timeout and output capture.
  • Search API — SerpAPI / Brave Search integration. Returns structured results with titles, URLs, and snippets.
  • File I/O — Read/write local files. Sandboxed to project directory. All writes require Cold Mode approval.
Connections

Above: Agents invoke tools via function calls. Below: Results fed to Knowledge layer for persistence. All tool calls are logged and timed.

# Tool decorator — registers with tool registry @tool("web_search") async def web_search(query: str) -> str: async with httpx.AsyncClient() as client: resp = await client.get(search_url, params={"q": query}) return resp.json()["results"][:5]

LLM Core

Ollama (Local) · Gemini · Claude
▸
What It Does

The brain. Provides inference for all agents. Falls back from local (Ollama) to cloud (Gemini/Claude) based on availability and task complexity. This layered approach gives us zero-cost local dev with cloud fallback for production.

Components
  • Ollama — Local inference, zero cost, llama3.2 default. Runs on CPU or GPU. Perfect for development and iteration.
  • Gemini — Cloud fallback via google-generativeai. Used when Ollama is too slow or task requires larger context window.
  • Claude — Anthropic API for complex reasoning. Extended thinking mode for multi-step analysis tasks.
Connections

Above: All layers call LLM for inference. This is the foundation everything builds on. Fallback chain: Ollama → Gemini → Claude.

# LLM with automatic fallback chain async def llm_call(prompt: str, model: str = "ollama"): for provider in [ollama, gemini, claude]: try: return await provider.generate(prompt) except ConnectionError: continue raise RuntimeError("All LLM providers failed")

1. User Input

A message arrives via Telegram, CLI, or API. The Gateway wraps it in a context object with user_id, timestamp, source metadata, and conversation history. This context travels with the message through the entire pipeline.

2. Gateway → Orchestrator

The Gateway normalizes the input and passes it to the Orchestrator. Cold Mode checks run here — if the user is asking for something destructive, approval is requested before proceeding. Timing starts.

3. Agent Router

Pattern matching selects the right agent. Simple queries go to Analyst. Web research to OSINT. Code execution to Executor. Complex tasks can spawn multiple agents working in parallel via the message bus.

4. Agent Execution

The selected agent runs its Think → Act → Observe loop. It may call tools (scraper, shell, search), consult memory for context, or request LLM inference for reasoning. Each step is logged with timing data.

5. Message Bus

Agent results are published to the message bus. Other agents can subscribe to topics. The Analyst aggregates results from OSINT + Executor when a task requires cross-agent collaboration.

6. Memory → Output

Results are persisted to SQLite (structured data) and ChromaDB (vector embeddings for future recall). The final output is formatted for the originating interface and sent back to the user. Timing stops.

What We Use vs What We Don't

Deliberate choices to keep the stack lean and maintainable

✓ Used in NEXUS-01

  • ✓ asyncio.Queue — In-process message bus, zero dependencies
  • ✓ SQLite — Zero-config persistent storage, single file
  • ✓ ChromaDB — Vector embeddings & semantic search
  • ✓ Ollama — Local LLM inference, zero cost
  • ✓ httpx + BeautifulSoup4 — Async web scraping
  • ✓ python-telegram-bot — Telegram interface
  • ✓ structlog — Structured, machine-readable logging

✗ Deliberately Avoided

  • ✗ Redis — External dependency, not needed at our scale
  • ✗ Supabase — Overkill for local-first development
  • ✗ LangGraph — Framework overhead for just 3 agents
  • ✗ Docker — Adds deployment complexity we don't need
  • ✗ Kubernetes — Way too early for this project
  • ✗ VPS — Runs locally first, cloud later
  • ✗ Prometheus/Grafana — structlog handles our observability
  • ✗ n8n — Overkill for our workflow needs
Philosophy Start simple. Add complexity only when the simple thing breaks.

Key Design Decisions

Every choice has a rationale and a trade-off we accept

Decision Rationale Trade-off
Ollama over paid APIs Zero cost during development, runs offline, no API keys needed Slower inference on complex prompts
SQLite over PostgreSQL Zero config, portable (single file), perfect for single-process No concurrent writes from multiple processes
asyncio.Queue over Redis No external dependencies, zero setup, in-process speed No persistence across restarts, no cross-process
3 agents not 30 Manageable complexity, easier debugging, clear responsibilities Less parallelism, single point of failure per agent
Python over Rust Rapid prototyping, massive AI ecosystem, team familiarity Slower execution, GIL limitations
CLI-first over Web Maximum control, fastest iteration, no frontend complexity Less visual, harder to demo to non-technical users
The Pattern Every "wrong" choice is actually a strategic decision to move faster now and optimize later. We choose speed of iteration over production readiness — and we know exactly when to switch.

Design Principles

Rules that govern every architectural decision

1. Local-First

Everything runs on your machine before it touches a network. SQLite for storage, Ollama for inference, asyncio for comms. Cloud is a fallback, not a requirement.

2. Explicit Over Implicit

No hidden state, no magic decorators, no framework-injected behavior. Every message, every tool call, every state transition is visible in the code.

3. Boring Technology

Use battle-tested tools. SQLite has been around since 2000. Python since 1991. asyncio since 2014. We don't need the latest framework to build something that works.

4. Delete Code Daily

If a feature isn't being used, delete it. If a layer can be merged, merge it. Complexity is the enemy — every line of code is a liability until it proves its value.
